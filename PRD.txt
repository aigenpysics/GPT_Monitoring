PRD: GPT 멀티 모니터 모니터링 대시보드
문서 버전: v1.0
작성일: 2026-03-15
문서 목적: GitHub Copilot이 바로 구현을 시작할 수 있을 정도로 구체적인 제품 요구사항과 구현 방향을 정의한다.

======================================================================
1. 제품 개요
======================================================================

제품명(가칭)
- GPT Monitor Wall
- GPT Multi-Workspace Dashboard

문제 정의
- 사용자는 여러 개의 GPT 대화창을 동시에 켜 두고 작업한다.
- 하지만 현재는 브라우저 탭/창을 계속 왔다 갔다 해야 해서 시간 낭비가 크다.
- 특히 어떤 GPT는 아직 답변 생성 중이고, 어떤 GPT는 이미 답변이 끝났는지 한눈에 알기 어렵다.
- 각 대화가 어떤 프로젝트인지, 다음에 무엇을 해야 하는지 맥락도 쉽게 잊어버린다.

제품 비전
- 두 번째 모니터를 전용 AI 작업 관제판처럼 사용한다.
- 사용자가 지정한 개수만큼 GPT 화면을 자동 분할해 동시에 보여 준다.
- 각 화면의 상태(생성 중 / 완료 / 로그인 필요 / 오류)를 시각적으로 표시한다.
- 답변이 끝난 화면은 눈에 띄게 깜빡여서 바로 확인할 수 있게 한다.
- 각 화면마다 "이 프로젝트가 무엇인지"와 "다음 액션이 무엇인지"를 같이 보여 준다.

핵심 가치
- 컨텍스트 스위칭 감소
- 멀티 태스킹 속도 향상
- 응답 완료 여부 즉시 파악
- 프로젝트 맥락 유지

대상 사용자
- 동시에 여러 GPT 대화를 돌리는 파워 유저
- 기획, 개발, 리서치, 문서 작성, 자동화 업무를 병렬로 진행하는 사용자
- 듀얼 모니터 이상 환경 사용자

MVP 범위
- 1차 구현은 ChatGPT 웹앱 모니터링에 집중한다.
- 구조는 확장 가능하게 설계하되, Claude/Gemini 등 타 서비스 지원은 후속 버전으로 미룬다.

비범위(Out of Scope) - MVP
- 모바일 앱
- 브라우저 확장 프로그램 버전
- 대화 자동 전송/자동 타이핑 에이전트 기능
- 여러 AI 서비스 동시 공식 지원
- 팀 협업, 클라우드 동기화
- 고급 분석 리포트

======================================================================
2. 제품 목표
======================================================================

목표
1) 사용자가 입력한 숫자만큼 GPT 화면을 자동으로 분할해서 한 창 안에 보여 준다.
2) 특정 화면을 전체화면(해당 앱 창 기준)으로 확대했다가 원래 분할 레이아웃으로 복귀할 수 있다.
3) GPT 답변 생성이 완료되면 해당 화면이 시각적으로 깜빡이며 완료 상태를 표시한다.
4) 각 화면에는 프로젝트명과 다음 액션을 보여 주는 가이드 패널이 존재한다.
5) 앱은 기본적으로 두 번째 모니터에서 열리며, 모니터 연결 상태를 기억한다.
6) 이전 작업 상태(화면 개수, URL, 메모, 프로젝트 정보, 마지막 레이아웃)를 저장하고 복원한다.

성공 지표(초기)
- 앱 실행 후 10초 이내에 마지막 워크스페이스 복원
- Pane(분할 화면) 4개 기준 부드럽게 상호작용 가능
- 답변 완료 알림을 사용자가 1초 이내에 시각적으로 인지 가능
- 사용자가 브라우저 탭 전환 없이 여러 GPT 작업을 병행 가능

3. 기본 가정 및 구현 방향
======================================================================

권장 구현 형태
- Electron 기반 데스크톱 앱
- 프론트엔드: React + TypeScript
- 상태 관리: Zustand 또는 Redux Toolkit
- 로컬 저장: electron-store 또는 SQLite
- 스타일: Tailwind CSS 또는 CSS Modules

권장 이유
- 외부 웹앱(ChatGPT)을 앱 내부에 여러 개 띄우기 쉬움
- 모니터 위치, 창 크기, 항상 위 옵션, 전체화면 제어가 쉬움
- 웹뷰(webview)를 각 pane에 넣어 독립적인 표시와 DOM 감시가 가능함

핵심 구현 전략
- 앱 내부에 N개의 pane을 만든다.
- 각 pane은 ChatGPT 웹사이트를 띄우는 webview를 가진다.
- 각 webview에는 preload script를 주입하여 DOM 변화를 관찰한다.
- DOM 변화와 버튼 상태를 기반으로 "생성 중/완료"를 추정한다.
- 프로젝트 정보/다음 액션은 pane 단위 메타데이터로 저장한다.

MVP 구현 전제
- ChatGPT의 내부 DOM 구조는 변경될 수 있으므로, 감지 로직은 하드코딩하지 말고 "사이트 어댑터" 구조로 만든다.
- ChatGPT용 selector/config는 별도 파일로 분리한다.
- 앱은 기본적으로 동일 계정 세션을 공유한다(로그인 한 번으로 모든 pane 사용 가능).

4. 사용자 시나리오
======================================================================

시나리오 A: 첫 실행
1) 사용자가 앱을 실행한다.
2) 앱은 연결된 모니터를 감지한다.
3) 외부 모니터가 있으면 해당 모니터에 창을 연다.
4) 사용자에게 "몇 개의 GPT 화면을 동시에 사용할지" 숫자 입력 UI를 보여 준다.
5) 사용자가 예: 4를 입력하면 2x2 그리드로 pane 4개가 배치된다.
6) 각 pane은 ChatGPT 홈 또는 마지막 사용 URL을 연다.

시나리오 B: 작업 중
1) 사용자는 pane 1에서 기획용 대화를, pane 2에서 코드용 대화를, pane 3에서 리서치용 대화를 연다.
2) 각 pane 상단에서 프로젝트명을 붙인다.
3) 오른쪽 가이드 패널에 다음 액션을 적는다.
4) 답변 생성이 끝난 pane은 테두리/배경이 깜빡이며 완료를 알린다.
5) 사용자는 완료된 pane만 골라서 읽는다.

시나리오 C: 확대/축소
1) 사용자는 pane 3의 전체화면 버튼을 누른다.
2) pane 3이 앱 창 전체를 차지한다.
3) 다른 pane은 숨겨지거나 최소 표시된다.
4) 사용자가 ESC 또는 축소 버튼을 누르면 원래 그리드로 복귀한다.

시나리오 D: 맥락 유지
1) 사용자는 여러 작업을 동시에 돌리다가 잠깐 자리를 비운다.
2) 돌아와서 각 pane 옆의 가이드 패널을 본다.
3) "이 프로젝트가 무엇인지"와 "다음에 할 일"을 빠르게 파악한다.
4) 바로 다음 작업을 이어서 진행한다.

6. 기능 요구사항 (Functional Requirements)
======================================================================

6.1 워크스페이스 생성 및 레이아웃

FR-001
- 앱은 사용자가 pane 개수를 숫자로 입력할 수 있어야 한다.
- 기본 허용 범위는 1~9개로 한다.
- 추후 12개까지 확장 가능하도록 내부 구조를 설계한다.

FR-002
- pane 개수 입력 즉시 그 수만큼 화면이 자동 분할되어야 한다.

FR-003
- 기본 배치 알고리즘은 다음 규칙을 따른다.
  - columns = ceil(sqrt(n))
  - rows = ceil(n / columns)
  - pane은 좌상단부터 우하단으로 채운다.

FR-004
- 레이아웃은 창 크기 변경 시 자동 재배치되어야 한다.

FR-005
- pane 개수를 변경하면 기존 pane의 URL/메타데이터를 최대한 유지해야 한다.
- pane 수를 줄일 때는 제거 대상 pane 데이터를 즉시 삭제하지 말고 임시 보관할 수 있다.

FR-006
- 사용자는 마지막 사용한 워크스페이스를 앱 재실행 시 자동 복원받아야 한다.

6.2 두 번째 모니터 동작

FR-007
- 앱 실행 시 연결된 디스플레이를 탐지해야 한다.

FR-008
- 외부 모니터가 있으면 기본적으로 외부 모니터(두 번째 모니터)에 앱 창을 열어야 한다.

FR-009
- 외부 모니터가 없으면 기본 모니터에 창을 열고, 설정에서 이후 외부 모니터 우선 옵션을 유지할 수 있어야 한다.

FR-010
- 앱은 마지막 창 위치, 크기, 어느 모니터에 있었는지 저장하고 복원해야 한다.

6.3 Pane 내 GPT 화면 표시

FR-011
- 각 pane은 독립적인 webview 영역을 가져야 한다.

FR-012
- 기본 URL은 ChatGPT 홈으로 설정한다.

FR-013
- 각 pane은 특정 대화 URL로 이동할 수 있어야 한다.

FR-014
- 모든 pane은 기본적으로 동일한 로그인 세션을 공유해야 한다.
- 즉, 사용자가 한 번 로그인하면 다른 pane에서도 같은 계정을 사용할 수 있어야 한다.

FR-015
- pane별 새로고침 기능이 있어야 한다.

FR-016
- pane별 상태가 눈에 띄게 표시되어야 한다.
- 최소 상태값:
  - idle
  - loading
  - generating
  - completed
  - completed-unread
  - login-required
  - error

6.4 전체화면 확대 / 축소

FR-017
- 각 pane에는 전체화면 확대 버튼이 있어야 한다.

FR-018
- 확대 시 선택한 pane만 앱 창 전체를 차지해야 한다.

FR-019
- 축소 버튼 또는 ESC 입력 시 원래 그리드 레이아웃으로 복귀해야 한다.

FR-020
- 확대 전 각 pane의 상태와 스크롤 위치는 가능한 한 유지해야 한다.

FR-021
- 한 번에 전체화면 pane은 하나만 허용한다.

6.5 답변 생성 완료 감지

FR-022
- 앱은 각 pane이 현재 GPT 응답 생성 중인지 감지해야 한다.

FR-023
- 앱은 생성 완료 시점을 감지해야 한다.

FR-024
- 생성 완료 감지는 preload script + DOM observer + 폴백 타이머 조합으로 구현해야 한다.

FR-025
- 최소 감지 규칙(초기 버전)은 다음을 따른다.
  1) 사용자가 메시지를 전송하거나 stop 버튼이 보이면 generating 상태로 전환
  2) assistant 응답 텍스트가 변경 중이면 generating 유지
  3) stop 버튼이 사라지고 응답 길이가 일정 시간(예: 1.5초) 더 이상 변하지 않으면 completed 상태로 전환
  4) completed 직후 아직 사용자가 해당 pane을 보지 않았으면 completed-unread 상태로 표시

FR-026
- 감지 로직은 사이트별 selector 설정 파일로 분리해야 한다.

FR-027
- selector가 실패할 경우 상태를 unknown/error로 기록하고 디버그 로그를 남겨야 한다.

6.6 완료 알림(깜빡임)

FR-028
- pane이 completed-unread 상태가 되면 해당 pane은 시각적으로 깜빡여야 한다.

FR-029
- 기본 깜빡임 동작은 다음과 같다.
  - 3회 반복
  - 총 3~5초 이내
  - pane 전체가 아니라 테두리 + 헤더 + 얕은 배경 하이라이트 중심

FR-030
- 사용자가 해당 pane을 클릭하거나 포커스하면 깜빡임을 중단하고 unread 상태를 해제해야 한다.

FR-031
- 깜빡임 효과는 설정에서 끌 수 있어야 한다.

FR-032
- 선택 기능으로 소리 알림 또는 OS 알림을 붙일 수 있으나, 기본값은 꺼 두어도 된다.

6.7 프로젝트 가이드 패널

FR-033
- 각 pane에는 프로젝트 맥락을 저장하는 Guide Panel이 있어야 한다.

FR-034
- Guide Panel은 기본적으로 pane 우측에 drawer 형태로 표시하거나 토글 가능해야 한다.

FR-035
- Guide Panel의 최소 필드는 다음과 같다.
  - projectName
  - summary
  - currentStatus
  - nextAction
  - notes
  - updatedAt

FR-036
- 사용자는 각 필드를 직접 수정할 수 있어야 한다.

FR-037
- Guide Panel 데이터는 로컬에 자동 저장되어야 한다.

FR-038
- pane이 너무 작아 Guide Panel을 항상 보여 주기 어려우면 compact overlay 모드로 자동 전환해야 한다.

FR-039
- compact overlay 모드에서는 최소한 다음 2개를 항상 볼 수 있어야 한다.
  - 프로젝트명
  - 다음 액션

FR-040
- Guide Panel에는 "자동 요약" 버튼을 둘 수 있다.
- MVP에서는 다음 중 하나로 구현한다.
  A안: 최근 대화 DOM을 읽어 간단 규칙 기반 요약 생성
  B안: 사용자가 별도 API 키를 넣었을 때만 외부 모델로 요약 생성
- 자동 요약이 어렵다면 우선 수동 입력 UX를 먼저 완성한다.

중요한 제품 판단
- "이 프로젝트가 무엇인지 / 다음에 뭘 해야 하는지"는 완전 자동보다 "수동 + 자동 보조"가 MVP에서 더 안정적이다.
- 따라서 반드시 수동 편집 가능해야 하며, 자동 기능은 보조 기능으로 설계한다.

6.8 상태 표시 및 시각화

FR-041
- 각 pane 헤더에는 상태 배지가 있어야 한다.

FR-042
- 상태별 시각 차이를 제공해야 한다.
  - loading: 로딩 인디케이터
  - generating: 움직이는 점 또는 진행 상태 표시
  - completed-unread: 강조 표시
  - login-required: 경고 표시
  - error: 에러 표시

FR-043
- 마지막 완료 시각(lastCompletedAt)을 pane에 기록해야 한다.

FR-044
- pane 헤더 또는 오버레이에 마지막 완료 후 경과 시간(예: 2m ago)을 선택 표시할 수 있다.

6.9 설정

FR-045
- 사용자는 다음 설정을 변경할 수 있어야 한다.
  - 기본 pane 개수
  - 외부 모니터 우선 여부
  - 항상 위(always on top)
  - 완료 시 깜빡임 on/off
  - 깜빡임 강도/횟수
  - 자동 복원 on/off
  - compact overlay 표시 여부
  - 디버그 모드 on/off

FR-046
- 설정은 로컬에 저장되어야 한다.

6.10 단축키

FR-047
- 최소 단축키를 제공한다.
  - F 또는 Enter: 활성 pane 확대
  - ESC: 원래 그리드 복귀
  - Ctrl/Cmd + R: 활성 pane 새로고침
  - Ctrl/Cmd + 1..9: 특정 pane 포커스
  - Ctrl/Cmd + Shift + G: Guide Panel 토글

6.11 장애/예외 처리

FR-048

FR-049

FR-050

7. 비기능 요구사항 (Non-Functional Requirements)

NFR-001 성능
- 4 pane 기준 일반적인 노트북/데스크톱 환경에서 무리 없이 동작해야 한다.

- pane 단위로 복구 가능해야 한다.

NFR-003 보안
- Node integration은 webview에서 비활성화한다.
- preload bridge를 최소 권한으로 설계한다.
- 로컬 저장 데이터에는 민감 정보(비밀번호, 토큰 원문)를 저장하지 않는다.

- Guide Panel 메모는 기본적으로 로컬 저장만 한다.

NFR-005 확장성

- 나중에 ChatGPT 외 사이트를 추가할 수 있게 구조화한다.

NFR-006 접근성
- 깜빡임 효과는 비활성화 가능해야 한다.
- 너무 강한 발광/빠른 점멸은 피한다.
- 최소 키보드 조작을 지원한다.

NFR-007 복원력
- 앱 비정상 종료 시 다음 실행에서 마지막 워크스페이스를 최대한 복원해야 한다.

10. 상태 머신 정의
======================================================================

Pane 상태 전이

idle -> loading
- URL 최초 로드 시작

loading -> idle
- 페이지 로드 완료, 아직 생성 중 아님

idle -> generating
- 사용자가 메시지를 보냈거나 stop 버튼 등장

generating -> completed
- stop 버튼 사라짐 + 응답 텍스트 안정화 + 일정 시간 변화 없음

completed -> completed-unread
- 완료 후 pane이 백그라운드이거나 사용자 포커스 없음

completed-unread -> idle
- 사용자가 pane 클릭/포커스

any -> login-required
- 로그인 UI 감지

any -> error
- webview crash, selector failure, load failure

비고
- 실제 구현에서는 completed와 completed-unread를 하나의 상태로 합치고 unread 플래그로 분리해도 된다.

13. 구현 세부 지침
======================================================================

13.1 레이아웃 엔진
- CSS Grid 기반으로 구현한다.
- paneCount에 따라 rows/columns를 계산한다.
- expandedPaneId가 있으면 해당 pane만 100% 영역 차지한다.
- 나머지 pane은 unmount하지 말고 hidden 처리 또는 behind 처리하여 상태를 최대한 보존한다.

13.2 webview 생성 정책
- 각 pane은 webview 태그를 사용한다.
- 공통 partition 예: persist:gpt-shared
- preload 경로를 지정한다.
- nodeIntegration=false, contextIsolation=true 원칙

13.3 상태 동기화
- preload -> renderer 통신 이벤트 예시
  - PANE_TITLE_UPDATED
  - PANE_STATUS_UPDATED
  - PANE_RESPONSE_PROGRESS
  - PANE_RESPONSE_COMPLETED
  - PANE_LOGIN_REQUIRED
  - PANE_ERROR

13.4 완료 감지 휴리스틱
- 텍스트 길이 추적 + 버튼 상태 + 시간 지연을 결합한다.
- 300~500ms 간격으로 마지막 assistant 응답 길이를 측정한다.
- 3회 연속 동일하고 stop 버튼이 없으면 complete 후보
- 이후 1.5초 debounce 후 완료 확정
- 완료 직후 lastCompletedAt 업데이트
- pane이 비활성 상태면 unread=true 및 blink trigger

13.5 프로젝트명/다음 액션 추출
- 우선순위
  1) 사용자가 직접 입력한 projectName, nextAction
  2) 자동 요약 결과
  3) 페이지 제목/최근 대화의 heuristic 추출
- 사용자가 수동 입력한 값은 자동 요약으로 덮어쓰지 않는다(manualOverride=true).

13.6 저장 정책
- Guide Panel 입력 후 500ms debounce 저장
- pane 상태는 중요한 이벤트 발생 시 저장
- 앱 종료 전 마지막 스냅샷 저장

15. 수용 기준 (Acceptance Criteria)
======================================================================

AC-001
Given 앱을 처음 실행했고 외부 모니터가 연결되어 있을 때
When 앱이 시작되면
Then 앱 창은 외부 모니터에 열려야 한다.

AC-002
Given 사용자가 pane 개수로 4를 입력했을 때
When 워크스페이스가 생성되면
Then 화면은 4개의 pane으로 자동 분할되어야 한다.

AC-003
Given 사용자가 특정 pane의 전체화면 버튼을 눌렀을 때
When 확대가 실행되면
Then 해당 pane은 앱 창 전체를 차지해야 한다.
And ESC 또는 축소 버튼으로 원래 레이아웃으로 복귀해야 한다.

AC-004
Given pane이 GPT 응답 생성 중일 때
When 응답이 종료되면
Then pane 상태는 completed 또는 completed-unread로 바뀌어야 한다.
And 사용자가 해당 pane을 보지 않았다면 시각적 깜빡임이 시작되어야 한다.

AC-005
Given 사용자가 completed-unread pane을 클릭했을 때
When pane이 포커스를 얻으면
Then 깜빡임은 멈추고 unread 상태가 해제되어야 한다.

AC-006
Given 사용자가 Guide Panel의 projectName과 nextAction을 입력했을 때
When 앱을 종료 후 다시 실행하면
Then 해당 값은 그대로 복원되어야 한다.

AC-007
Given pane 폭이 매우 작을 때
When Guide Panel을 항상 표시하기 어려우면
Then compact overlay로 자동 전환되어 프로젝트명과 다음 액션이 보여야 한다.

18. Copilot에게 전달할 구현 원칙
======================================================================

아래 원칙을 우선 적용해서 코드를 생성한다.

1) 코드는 TypeScript 기준으로 작성한다.
2) Electron 메인/렌더러/프리로드를 명확히 분리한다.
3) pane 로직은 재사용 가능한 컴포넌트로 분리한다.
4) 사이트 감지 로직은 adapter 패턴으로 분리한다.
5) 상태 이름과 이벤트 이름은 사람이 읽기 쉽게 명확히 짓는다.
6) 로컬 저장 구조는 JSON 직렬화가 쉬운 형태로 유지한다.
7) DOM selector는 코드 상수에 박아 넣지 말고 별도 설정 파일로 관리한다.
8) UI보다 먼저 상태/이벤트/저장 구조를 안정적으로 만든다.
9) 크래시가 나도 앱 전체가 죽지 않도록 pane 단위 실패를 격리한다.
10) MVP에서는 자동 요약보다 수동 메모 UX를 우선한다.

19. Copilot 시작 프롬프트 예시

다음 요구사항의 Electron + React + TypeScript 데스크톱 앱을 만들어라.

- 앱은 외부 모니터가 있으면 그 모니터에 열린다.
- 사용자가 입력한 숫자(1~9)만큼 pane을 CSS Grid로 생성한다.
- 각 pane은 ChatGPT를 띄우는 webview를 가진다.
- pane 헤더에는 제목, 상태 배지, 새로고침 버튼, 전체화면 버튼, Guide Panel 토글 버튼이 있다.
- 전체화면 버튼을 누르면 해당 pane이 앱 창 전체를 차지하고, ESC로 원래 그리드로 복귀한다.
- preload script로 ChatGPT DOM을 감시하여 generating/completed 상태를 추정한다.
- completed-unread 상태가 되면 pane 테두리와 헤더가 3회 깜빡인다.
- 각 pane에는 projectName, summary, currentStatus, nextAction, notes를 저장하는 Guide Panel이 있다.
- Guide Panel 데이터와 워크스페이스 상태는 로컬 파일로 자동 저장/복원한다.
- 구조는 adapter 패턴으로 설계하여 나중에 다른 AI 사이트도 추가 가능하게 한다.
- nodeIntegration은 끄고 preload bridge만 사용한다.
- 코드 구조는 main / renderer / preload 폴더로 분리한다.
- 먼저 동작 가능한 MVP를 만들고, 이후 자동 요약 기능은 TODO로 남겨라.

======================================================================
20. 최종 판단
======================================================================

이 제품의 MVP 핵심은 아래 4가지다.
1) 멀티 pane으로 GPT를 한 화면에 띄우기
2) pane 확대/축소
3) 답변 완료 시 시각 알림
4) 프로젝트명/다음 액션 메모 유지

이 4가지만 제대로 되면 사용자는 이미 브라우저 탭 전환 비용을 크게 줄일 수 있다.
자동 요약/고급 분석은 후속 기능으로 미뤄도 제품 가치는 충분하다.
