1. 개발 방식
======================================================================

원칙
- 작은 PR 단위로 진행한다.
- 매 단계마다 실행 가능한 상태를 유지한다.
- UI보다 상태머신과 데이터 흐름을 먼저 안정화한다.
- PostHog는 초기에 "이벤트 잘 쌓이는지"만 검증하고, 대시보드는 나중에 정리한다.
- embedded ChatGPT에 analytics/autocapture 스크립트를 넣지 않는다.
- Main Process를 시스템 오케스트레이터로 본다.

4. 구현 순서 (중요)
======================================================================

Milestone 1: 앱 부트스트랩
목표
- Electron 창이 뜬다.
- React HUD가 보인다.
- 외부 모니터가 있으면 거기에 띄운다.

완료 조건
- npm run dev로 앱이 실행된다.
- 모니터 정보가 콘솔에 표시된다.

Milestone 2: PaneManager + Grid
목표
- 사용자가 pane 개수를 정하면 WebContentsView가 그 수만큼 생성된다.
- 자동 grid가 계산된다.

완료 조건
- 1~9 pane 생성 가능
- 각 pane bounds가 정상 배치
- pane마다 ChatGPT 로드 가능

Milestone 3: ChatGPT observer v1
목표
- preload에서 ChatGPT 상태를 감지한다.
- generating / completed / unknown 정도만 먼저 처리한다.

완료 조건
- 상태 배지가 HUD에 반영된다.
- 완료 시점에 pane completed 이벤트가 내부적으로 발생한다.

Milestone 4: completed-unread + Ready Queue
목표
- 완료되었지만 아직 확인하지 않은 pane을 유지한다.
- Ready Queue를 붙인다.

완료 조건
- pane을 안 보고 있을 때 완료되면 Ready Queue에 뜬다.
- pane 확인 시 unread가 해제된다.

Milestone 5: Guide Panel + Local persistence
목표
- pane별 projectName / nextAction / priority를 저장한다.
- 재실행 후 복원된다.

완료 조건
- guide 입력 후 앱 재시작해도 남아 있다.

Milestone 6: PostHog telemetry
목표
- Local queue + PostHogSink를 만든다.
- opt-in일 때만 전송한다.

완료 조건
- workspace opened, pane completed, pane marked read, guide updated, preset applied가 PostHog에 들어간다.
- 잘못된 token이어도 앱은 계속 동작한다.

Milestone 7: Settings + Feature flags scaffold
목표
- telemetry opt-in/settings UI
- PostHog feature flag fetch 틀

완료 조건
- host/token 저장 가능
- feature flag fetch 실패가 UX를 막지 않음

Milestone 8: hardening
목표
- slow/stalled/error/login-required
- retry worker
- shutdown flush
- keyboard shortcuts
- tests

완료 조건
- 기본 QA 시나리오 모두 통과

5. GitHub Copilot에 바로 넣을 작업 프롬프트
======================================================================

Prompt A: 부트스트랩
"Electron + React + TypeScript 프로젝트를 생성해줘. Main Process, Renderer, Preload, Shared 폴더를 분리하고, 외부 모니터가 있으면 거기에 BrowserWindow를 띄우는 코드까지 만들어줘. 코드 스타일은 엄격한 TypeScript와 함수 분리 중심으로 해줘."

Prompt B: PaneManager
"Electron Main Process에서 WebContentsView 여러 개를 생성하고, 사용자가 입력한 paneCount(1~9)에 따라 자동 grid layout으로 배치하는 PaneManager 클래스를 만들어줘. BrowserView는 쓰지 말고 WebContentsView를 사용해줘."

Prompt C: ChatGPT observer
"Preload script에서 ChatGPT DOM을 관찰해 generating/completed/unknown 상태를 추정하는 ChatGPTAdapter를 만들어줘. DOM selector 실패나 페이지 구조 변경이 있어도 예외를 던지지 말고 unknown으로 폴백해줘. 상태 변화는 IPC로 Main Process에 보내줘."

Prompt D: Ready Queue
"Renderer에서 completed-unread pane을 모아 보여주는 Ready Queue 컴포넌트를 만들어줘. queue item 클릭 시 pane focus IPC를 호출하고 unread를 해제해줘."

Prompt E: Local persistence
"workspace, panes, guides, presets, readyQueue를 SQLite 또는 단순 로컬 저장소로 저장/복원하는 Repository와 Service 계층을 만들어줘. 타입은 zod로 검증해줘."

Prompt F: PostHog telemetry
"Electron Main Process용 TelemetryService를 만들어줘. Local queue에 이벤트를 먼저 저장하고, opt-in일 때만 posthog-node로 전송해줘. PostHog 전송 실패는 앱을 망가뜨리면 안 되고, queue retry worker가 나중에 재전송하게 해줘."

Prompt G: privacy
"Prompt/response 원문 텍스트는 텔레메트리로 보내지 않도록 eventFactory에서 스키마를 강제해줘. 자유 텍스트 필드는 hash나 length만 허용해줘."

Prompt H: feature flags
"Main Process에서만 PostHog feature flags를 불러오는 FeatureFlagService를 만들어줘. blinkEnabled, slowThresholdMs, stalledThresholdMs 같은 원격 설정 키를 지원하고, 실패 시 로컬 기본값으로 동작하게 해줘."

8. PostHog 구현 가이드
======================================================================

기본 전략
- posthog-node는 Main Process에서만 쓴다.
- Renderer/Preload는 IPC로 Main에 domain event를 보낸다.
- TelemetryService가 eventFactory를 통해 정규화한 후 capture한다.

전송 모드
- 개발 초기:
  - Local only
- PostHog 연결 후:
  - Local + PostHog
- fallback:
  - PostHog HTTP API transport는 나중에 추가

꼭 지킬 규칙
- telemetryOptIn이 false면 원격 capture 금지
- prompt/response 본문 금지
- notes/objective 원문 금지
- projectName은 hash만 허용
- 앱 종료 시 flush/shutdown 호출
- 전송 실패는 로그만 남기고 UX는 계속

12. ChatGPT observer v1 구현 힌트
======================================================================

v1 목적
- 100% 정확도가 아니라 "안전한 추정"이 목표다.
- generating/completed를 우선 잡는다.

가능한 신호
- stop 버튼 존재
- assistant message 수 증가
- DOM mutation이 계속 발생
- 일정 시간 mutation 정지
- 로그인 화면 마커
- 오류 페이지 마커

중요
- selector는 한 군데만 믿지 말고 복수 후보를 둔다.
- heuristic 신뢰도가 낮으면 unknown.
- DOM 구조 변경은 자주 일어날 수 있다고 가정한다.
